# Warning `#2`: [`mount()`](/api/view-models/interface#mount-void-promise-void) returned a `Promise` during **server** render

## Full text

`ViewModel.mount()` returned a Promise during server render.  
HTML is emitted before the promise settles; the first server string may not reflect post-mount state (for example views using [`isAbleToRenderView`](/api/view-model-store/interface#isabletorenderviewviewmodelid)).

## Explanation

On the server, React renders in a single pass. When [`attach()`](/api/view-model-store/interface#attachviewmodel) runs [`ViewModelStoreBase.attach`](/api/view-model-store/base-implementation), a **synchronous** `mount()` completes before the component tree continues, so SSR and the first client paint can match.

If `mount()` returns a Promise, [`attach()`](/api/view-model-store/interface#attachviewmodel) returns that promise too; the string from `renderToString` / streaming may be produced **before** the promise resolves. That is expected and often acceptable (e.g. loading states). It is **not** a problem on the client in the same way, which is why this warning is only logged in development when `window` is undefined (typical Node SSR).

See also: [SSR and async `mount()`](/react/ssr#async-mount).

## Potential solutions

- Prefer **synchronous** `mount()` for anything that must affect the first server HTML, and move async work to `useEffect` on the client or to explicit data loading before render.
- Or rely on a **fallback** UI until `mount()` completes, keeping server and client output consistent (see [SSR docs](/react/ssr)).
